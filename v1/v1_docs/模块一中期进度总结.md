# 模块一中期进度总结

## 一、技术路线

模块一当前采用 Claude Agent SDK 风格的 Agent 架构。整体设计思路是：大模型负责决策，后端工具负责执行。

也就是说，Agent 不是一次性回答用户问题，而是按照“LLM 决策 + 工具执行 + 结果回传 + 再决策”的方式运行。用户输入一个事件后，系统先确定事件范围，再启动新闻 Agent。新闻 Agent 会让 LLM 判断应该搜索哪些关键词、查哪些权威来源，然后调用 `web_search` 工具进行真实网页搜索。搜索结果返回后，LLM 再选择需要抓取的网页，调用 `fetch_url` 工具获取正文，最后把新闻来源、正文、新闻事件块等内容整理成结构化的事件基础信息库。

当前技术链路可以概括为：

```text
用户输入事件
→ 统筹逻辑生成事件范围和采集任务
→ 新闻 Agent 由 LLM 决策搜索方向
→ LLM 调用 web_search
→ 后端通过 Tavily 执行真实网页搜索
→ 返回候选新闻 URL
→ 来源白名单过滤
→ LLM 选择需要抓取的 URL
→ 后端抓取网页正文
→ 生成结构化新闻材料
→ 写入事件基础信息库
```

当前架构中，各部分职责如下：

```text
LLM：负责理解用户输入、规划搜索方向、决定调用哪些工具
Python 后端工具：负责执行 web_search、fetch_url、来源过滤、落盘等操作
Tavily：提供真实网页搜索能力
网页抓取器：负责抓取候选 URL 的正文内容
SourceRegistry：负责来源白名单过滤，避免未知来源污染材料层
StorageWriter：负责把最终结构化结果写入本地事件基础信息库
```

当前搜索能力已经从早期的 RSS 调整为真实网页搜索。`web_search` 目前接入 Tavily API，可以支持历史事件和当前事件的网页检索。

模块一第一版的目标不是直接生成最终分析报告，而是先形成一个可靠的事件基础信息库，为后续新闻分析、金融分析和事件地图构建提供材料基础。

## 二、已实现内容

目前模块一第一版 MVP 已经跑通，已经具备从用户输入事件到采集新闻材料并结构化落盘的完整闭环。

已实现的能力包括：

1. 用户可以在前端输入事件，例如“2026 年美伊冲突”。
2. 后端能够启动模块一采集流程。
3. 新闻 Agent 能够调用大模型进行多轮决策。
4. 大模型能够决定搜索方向，并调用 `web_search` 工具。
5. `web_search` 已经接入 Tavily，能够执行真实网页搜索。
6. 搜索结果会经过来源白名单过滤，避免未知网站、博客或低质量来源进入材料层。
7. Agent 能够调用 `fetch_url` 抓取候选新闻网页正文。
8. 抓取成功的网页正文会保存到本地。
9. 系统能够生成事件基础信息库，包括新闻来源、新闻事件块、证据引用等结构化字段。
10. 每一次运行都会保存完整过程记录，便于后续调试和复盘。

最近一次测试中，输入事件为：

```text
2026 年美伊冲突
```

系统成功完成运行，结果如下：

```text
状态：完成
错误：无
新闻来源数量：5
新闻事件块数量：1
证据引用数量：5
```

本次成功采集到的来源包括 AP News 和 Al Jazeera。所有成功进入事件基础信息库的来源都显示为 `tavily_search`，说明搜索链路已经真正走通了 Tavily 的网页搜索，而不是旧的 RSS 或静态假数据。

本次运行证明了几个关键点：

1. LLM 能够调用搜索工具。
2. Tavily 能够返回真实候选网页。
3. 系统能够对白名单来源进行筛选。
4. 系统能够抓取部分新闻网页正文。
5. 最终结果可以写入本地事件基础信息库。

当前每次运行都会保存 trace。trace 中可以看到 Agent 每一步做了什么，包括 LLM 的决策轮次、`web_search` 调用、Tavily 返回候选数量、`fetch_url` 抓取成功或失败、最终落盘状态等。这对后续排查“为什么没有结果”“为什么抓错来源”“为什么某个网页失败”非常重要。

当前测试结果已经通过，说明第一版主流程具备基本稳定性。

```text
36 passed
```

## 三、未实现内容与当前问题

虽然模块一第一版已经跑通，但当前还只是 MVP 阶段，仍有一些未实现内容和明显问题。

第一，当前架构是 Claude Agent SDK 风格的自实现工具调用框架。模型接口使用 Anthropic / Claude 兼容格式，但工具循环主要由我们自己的 Python 后端控制。后续如果需要完全迁移到官方 Claude Agent SDK Runtime，还需要进一步封装。

第二，统筹 Agent 还比较初级。当前系统可以基于用户输入生成基础事件范围，但还没有完整实现多轮用户确认、模糊事件消歧、事件范围确认卡片等能力。例如用户只输入“美伊冲突”时，系统未来应该能够判断存在多个不同时间段的美伊冲突，并让用户确认具体要分析哪一次。

第三，基准事件时间线还没有真正实现。当前时间线仍然比较粗，主要是一个基础背景节点。后续需要接入 Wikipedia、Wikidata 或其他百科数据源，让统筹 Agent 先生成更可靠的基准时间线，再让新闻 Agent 围绕时间线节点采集材料。

第四，新闻正文清洗还不够好。当前抓取网页正文时，有时会混入导航栏、菜单、栏目名称、推荐新闻等网页噪声。这会影响后续新闻块摘要质量。后续需要优化正文抽取和页面清洗能力。

第五，LLM 的结构化输出还不够稳定。最近一次运行中，大模型某一轮返回了格式不合法的 JSON，系统虽然通过 fallback 机制继续完成了任务，但说明后续需要增加 JSON 修复重试机制，减少结构化输出失败的情况。

第六，新闻事件块质量还比较初级。当前系统已经能生成 `news_blocks`，但摘要和多来源综合能力还不够好。后续应该让 LLM 基于多个来源进行更准确的归纳，区分事实、来源差异和时间节点，而不是简单截取正文片段。

第七，金融 Agent 还没有实现。目前金融部分只是占位，还没有真正抓取油价、黄金、美元指数、股指、航运、国防股等市场数据，也没有生成金融证据或市场趋势区间。

第八，事件地图 Agent 暂未实现。当前第一版只做到事件基础信息库生成，尚未把新闻、金融、时间线统一组织成最终的 context map。

第九，来源覆盖还需要扩展。当前主要依赖 AP News、Al Jazeera、Reuters、BBC 等白名单来源。后续可以继续加入更多政府公告、国际组织、地区媒体、当地媒体、智库和金融媒体。

第十，网页抓取仍会受到反爬限制。部分新闻网站，例如 Reuters，可能出现 401 Forbidden 或 SSL 读取失败。当前系统可以跳过失败来源并继续运行，但如果后续要提升覆盖率，需要增强抓取能力，例如支持更好的网页正文抽取、重试策略，甚至浏览器级抓取。

阶段性结论是：

```text
模块一已经完成第一版可运行闭环，能够从用户输入事件出发，通过 LLM Agent 调用真实网页搜索工具，采集权威新闻来源，抓取正文，并生成本地事件基础信息库。
```

下一阶段重点应该从“跑通流程”转向“提高质量”，主要包括：完善统筹 Agent 的事件范围确认能力，接入百科时间线，优化网页正文清洗，提高新闻事件块质量，并开始设计金融 Agent。
