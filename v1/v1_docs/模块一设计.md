# 模块一设计：现实基础信息层

## 一、模块定位

模块一是 HelloWorld 的现实基础信息层，负责把用户输入的现实事件整理成后续推演可以使用的基础材料。

它不是一个单纯的新闻爬虫，也不是一个简单的搜索工具，而是整个系统的现实信息底座。模块一需要围绕一个事件收集新闻事实、公开资料、金融市场信号、预测市场信息和历史市场反应，并把这些材料整理成结构化、可引用、可追溯的事件材料包。

模块一要回答三个问题：

1. 现实世界中发生了什么？
2. 金融市场或预测市场如何理解这件事？
3. 这些新闻材料和金融材料之间是什么关系，不同来源的报道差异和材料缺口在哪里？

模块一第一版的产物是事件基础信息库。它先把用户确认后的事件范围、基准时间线、新闻材料、金融占位结果、原始来源和结构化摘要保存下来。模块二或后续版本可以基于这份事件基础信息库继续生成 Context Map、推演上下文或问答视图，但第一版不把 Context Map 作为验收目标。

## 二、最小设计目标

第一版模块一不追求完整的实时新闻监控系统，也不急于做复杂知识图谱。最小目标是打通一条清晰链路：

```text
用户输入事件
  ↓
事件接入与任务分发 Agent
  ↓
基准事件时间线
  ↓
新闻 Agent + 金融 Agent
  ↓
事件基础信息库
```

用户输入一个事件后，例如“美伊冲突”，模块一需要先通过事件接入与任务分发 Agent 确定事件范围，并把确认后的范围展示给用户。用户确认范围无误后，统筹 Agent 先从 Wikipedia、Wikidata、百科页面或其他概览资料中提取一条基准事件时间线，再围绕这条时间线启动新闻 Agent，并为后续金融 Agent 保留任务接口。新闻 Agent 整理现实材料，金融 Agent 第一版可以返回空占位结果，两者结果共同进入事件基础信息库。只要事件基础信息库完整落盘，第一版模块一就算成功。

## 三、为什么需要事件接入与任务分发 Agent

用户输入往往是模糊的。比如用户只输入“美伊冲突”，这个短语可能指向很多不同事件范围：

- 当前正在发生的新一轮美伊紧张局势；
- 2020 年苏莱曼尼事件；
- 伊朗核问题；
- 霍尔木兹海峡、伊拉克、叙利亚或红海相关冲突；
- 一个假设问题，例如“如果美伊冲突升级”；
- 一个金融问题，例如“美伊冲突对油价有什么影响”。

如果直接把“美伊冲突”丢给新闻 Agent 和金融 Agent，它们可能会各自形成不同理解。新闻 Agent 可能查当前新闻，金融 Agent 可能分析历史油价，最后两类材料无法很好对齐。

因此，模块一需要在新闻 Agent 和金融 Agent 之前加入一个轻量统筹角色：事件接入与任务分发 Agent。

它不负责最终研判，也不替代新闻 Agent 或金融 Agent 做检索分析。它的作用是先和用户一起确定事件范围，避免后续 Agent 查偏；在用户确认范围后，再读取百科类概览资料提取基准事件时间线，并把这条时间线转化为新闻 Agent 和金融 Agent 可以执行的时间线驱动任务。

## 四、模块一核心角色

模块一第一版包含三个活动角色。事件地图 Agent 暂不实现，放到后续版本。

| 角色 | 出现时机 | 核心问题 | 输出 |
| --- | --- | --- | --- |
| 事件接入与任务分发 Agent | 新闻 Agent 和金融 Agent 启动前 | 用户到底想查哪个事件范围？事件主线怎么展开？新闻和金融各自该查什么？ | 事件范围确认卡、基准事件时间线、时间线驱动采集任务单 |
| 新闻 Agent | 时间线驱动任务生成后 | 现实中发生了什么？基准时间线有没有漏掉重要节点？ | 新闻事件块、来源引用、报道差异、时间线修正建议 |
| 金融 Agent | 时间线驱动任务生成后 | 市场如何理解这件事？ | 第一版只保留任务接口和空占位结果，后续版本再输出金融证据 |

这三个角色的边界要保持清楚。事件接入与任务分发 Agent 负责“确定范围、提取基准时间线和分配任务”，新闻 Agent 和金融 Agent 负责“围绕时间线收集和整理材料”。第一版不负责结合材料生成最终事件结构，也不负责 Context Map、基础材料问答、多主体推演或最终报告。

## 五、事件接入与任务分发 Agent 设计

### 1. 核心职责

事件接入与任务分发 Agent 负责接收用户输入，先确定事件范围，再生成基准事件时间线和时间线驱动采集任务单。

它主要做八件事：

1. 规范化用户输入；
2. 初步查找或识别可能对应的候选事件范围；
3. 如果存在多个可能事件，与用户对话确认要查哪一个；
4. 向用户返回事件范围确认卡，让用户确认或修改；
5. 在用户确认事件范围后，读取 Wikipedia、Wikidata、百科页面或其他概览资料；
6. 从概览资料中提取按时间排列的基准事件时间线；
7. 把时间线转化为新闻 Agent 和金融 Agent 的采集任务；
8. 规定两个 Agent 必须返回的结构化字段。

例如用户输入：

```text
美伊冲突
```

事件接入与任务分发 Agent 不应该直接默认它指向“当前事件”。它应该先识别这个输入可能对应的多个事件范围，并向用户确认。例如：

```text
我找到多个可能的“美伊冲突”事件范围：

1. 2020 年苏莱曼尼事件及其后续美伊紧张升级
2. 近期美伊围绕中东地区冲突、核问题或代理人武装的紧张局势
3. 霍尔木兹海峡、伊拉克、叙利亚或红海相关的美伊间接冲突

请确认你想分析哪一个范围，或补充具体时间、地区、关键词。
```

如果用户明确输入“2020 年美伊冲突”，统筹 Agent 不需要把它额外分类成“历史事件”，只需要把事件范围锁定在 2020 年相关美伊冲突上。后续新闻 Agent 不应去查 2018 年的美伊事件并把它当作本轮事件；但可以查找 2020 年对该事件的总结分析、时间线、复盘文章和市场反应材料。较早年份的材料只能作为背景补充，并且必须明确标注为背景。

### 2. 模糊输入处理原则

模块一遇到模糊输入时，应优先确认事件范围，而不是直接做默认假设。

如果用户输入“美伊冲突”这类宽泛短语，事件接入与任务分发 Agent 应先进行轻量范围识别，判断它可能对应哪些事件、时间段、地区或议题。只要发现多个合理候选范围，就应该向用户追问，让用户选择或修正。

追问不需要很复杂，可以直接给出候选项：

```text
你说的“美伊冲突”可能指多个事件范围。请确认你想查哪一个：

1. 2020 年苏莱曼尼事件及后续冲突
2. 最近一轮美伊紧张局势
3. 伊朗核问题相关的美伊对立
4. 霍尔木兹海峡或中东代理人冲突相关事件
5. 其他，请补充时间、地区或关键词
```

用户选择或补充范围后，统筹 Agent 还需要生成一条事件范围确认卡，让用户最后确认。只有用户确认后，新闻 Agent 和金融 Agent 才能开始执行。

如果用户给出的输入已经非常明确，例如“2020 年 1 月苏莱曼尼遇袭后的美伊冲突”，统筹 Agent 可以不再列候选项，但仍应返回事件范围确认卡，让用户确认是否查这个范围。

### 3. 事件范围确认卡

事件接入与任务分发 Agent 在启动新闻 Agent 和金融 Agent 前，必须向用户返回一条事件范围确认卡。

确认卡用于告诉用户：系统准备按照什么范围去查。如果范围不对，用户可以修改；如果范围正确，用户确认后再执行。

确认卡可以包含：

| 字段 | 说明 |
| --- | --- |
| 事件名称 | 系统规范化后的事件名称 |
| 事件内容 | 对本轮要分析事件的简短说明 |
| 时间范围 | 本轮检索和分析聚焦的时间段 |
| 地理范围 | 主要涉及地区 |
| 关键主体 | 国家、组织、人物、企业或市场主体 |
| 关键词 | 后续新闻和金融检索会使用的关键词 |
| 排除范围 | 明确不纳入本轮分析的相似事件 |
| 背景材料规则 | 哪些旧材料只能作为背景，不能当作本轮事件 |
| 待用户确认 | 用户确认、修改或重新选择 |

例如：

```text
事件范围确认卡

事件名称：2020 年苏莱曼尼事件后的美伊冲突
事件内容：围绕 2020 年 1 月美军击杀伊朗将领苏莱曼尼及后续伊朗报复、美国回应、地区局势和市场反应展开整理。
时间范围：主要聚焦 2020 年 1 月前后，允许补充 2020 年内的复盘和分析材料。
地理范围：伊拉克、伊朗、美国、中东地区。
关键主体：美国、伊朗、伊拉克、伊斯兰革命卫队、美军、国际组织。
关键词：Soleimani, Qasem Soleimani, Iran US conflict 2020, missile retaliation, oil price reaction。
排除范围：不把 2018 年美伊关系事件或其他年份的冲突当作本轮事件主体。
背景材料规则：2018 年以前的材料只能作为背景，不进入本轮事件事实线。

请确认是否按这个范围执行。
```

只有用户确认事件范围后，统筹 Agent 才能继续生成基准事件时间线和时间线驱动采集任务单。基准时间线本身不需要再单独让用户确认，否则流程会过重；如果后续新闻 Agent 或金融 Agent 发现时间线有遗漏或错误，第一版只把修正建议写入事件基础信息库，不在本轮统一生成最终时间线。

### 4. 不应该负责的事情

事件接入与任务分发 Agent 不应该做以下事情：

- 不直接生成最终分析报告；
- 不替代新闻 Agent 检索和整理新闻；
- 不替代金融 Agent 分析市场信号；
- 不决定模块三要启动哪些推演 Agent；
- 不进行美伊双方行动预测；
- 不把新闻事实、市场信号和模型推测混在一起下结论。

它的边界是：

```text
负责和用户确认事件范围，并把确认后的范围变成一条基准事件时间线和模块一内部可执行的时间线驱动采集任务。
```

## 六、基准事件时间线与时间线驱动任务

用户确认事件范围后，事件接入与任务分发 Agent 的重点不是直接把任务分给新闻 Agent 和金融 Agent，而是先生成一条基准事件时间线。

基准事件时间线是本轮事件的初始骨架。它可以来自 Wikipedia、Wikidata、百科页面、专题页、已有年表或其他概览资料。不同事件页面的组织方式不一样，有些按日期写，有些按阶段写，有些只给出概述。统筹 Agent 不能照搬页面标题，而要尽量把内容整理成按天排列的事件时间线。

基准事件时间线不是最终事实结论。它只是帮助后续 Agent 不查偏的初始导航。新闻 Agent 和金融 Agent 会围绕这条时间线采集证据；第一版不会再整理最终时间线，只保存基准时间线、时间线修正建议和相关来源，供后续版本处理。

### 1. 基准时间线的时间表达

基准时间线不能只用抽象节点，例如“升级”“回应”“降温”。它应尽量按照真实时间顺序组织，默认以“天”为基本粒度，使用明确的年月日。只有当事件本身持续多天、资料本身只能确定区间，或金融变化表现为跨日期趋势时，才使用日期区间。

时间线可以包含两类时间项：

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| 点状事件 | 发生在某一天或某个明确时间点的关键事件 | 2026-02-28 某次袭击发生 |
| 区间事件 | 持续多天的阶段、趋势或连续行动 | 2026-03-01 至 2026-03-03 连续空袭与回应 |

新闻材料通常更容易挂到点状事件上，因为一条新闻往往对应某天发生的动作、声明或袭击。金融材料则既可以挂到点状事件，也可以挂到区间事件。例如油价上涨、避险资产走强或航运风险溢价上升，可能不是一天完成的，而是在某个时间窗口内持续形成。这类金融变化应作为“市场趋势区间”挂到时间线上，而不是被强行压成单个节点。

### 2. 基准时间线的节点结构

基准时间线中的每个时间项可以包含：

| 字段 | 说明 |
| --- | --- |
| timeline_item_id | 时间项 ID |
| time_type | point、interval 或 background |
| start_date | 开始日期 |
| end_date | 结束日期；点状事件可以为空或等于 start_date |
| title | 时间项标题 |
| summary | 时间项摘要 |
| actors | 涉及主体 |
| locations | 涉及地区 |
| source_hint | 初始来源，例如 Wikipedia 页面、Wikidata、百科年表 |
| search_keywords | 后续新闻和金融检索的起始关键词 |

这里暂时不要求统筹 Agent 判断某个节点一定是核心节点、辅助节点还是背景节点，也不要求它规定这个节点必须补充哪些材料。统筹 Agent 的信息来源主要是百科概览，它还没有新闻 Agent 和金融 Agent 后续补充的真实证据，因此不适合过早决定节点的重要性或证据方向。

`search_keywords` 只是起始检索线索，不是限制条件。新闻 Agent 和金融 Agent 可以根据实际搜索结果扩展关键词、补充相关材料，不能只按统筹 Agent 给出的关键词机械检索。

### 3. 背景与主线的关系

基准时间线可以保留背景，但背景不能混入本轮事件事实线。

例如用户确认分析某个 2026 年事件时，统筹 Agent 可以把事件开始前的长期冲突、历史协议、制裁背景或地区矛盾整理为 background 类型时间项。背景项可以帮助新闻 Agent 和金融 Agent 理解上下文，但不能被当成本轮事件中的新闻节点，也不能和本轮市场反应混在一起。

因此，时间线可以采用下面这种结构：

```text
背景：事件开始前的长期矛盾、历史协议、制裁或地区局势
2026-02-28：关键袭击、领导人死亡、正式开战或其他核心事件
2026-03-01：某一方发动报复行动
2026-03-02：另一方发布回应或扩大行动
2026-03-03 至 2026-03-05：连续打击、外交斡旋或市场趋势形成
2026-03-10：协议、停火、谈判破裂、封锁或其他关键后续节点
后续日期：停火、协议、谈判破裂、封锁、制裁或其他后续节点
```

这类主线才适合后续前端展示。用户看到的是按照日期展开的事件发展，而不是一组抽象的分析标签。以周为单位的表达只能作为某些长事件页面的原始材料来源，不应成为系统默认的时间线粒度。

### 4. 时间线驱动采集任务单

基准事件时间线生成后，统筹 Agent 会把它转化为时间线驱动采集任务单。

这份任务单不是最终报告，而是给新闻 Agent 和金融 Agent 的内部执行说明。它的目的有三个：

1. 让两个 Agent 围绕同一条事件时间线工作；
2. 让每条新闻和每个金融信号都尽量挂回某个时间项；
3. 避免把相似但不属于本轮范围的事件混进来。

第一版任务单可以先保持最小范围，只保留能让新闻 Agent 和金融 Agent 开始工作的必要字段：

| 字段 | 说明 |
| --- | --- |
| event_query | 用户原始输入 |
| confirmed_event | 用户确认后的事件名称 |
| confirmed_scope | 用户确认后的事件范围说明 |
| baseline_timeline | 基准事件时间线，包含点状事件、区间事件和背景项 |
| exclude_scope | 明确不纳入本轮分析的相似事件或时间范围 |
| news_task | 新闻 Agent 的任务说明 |
| finance_task | 金融 Agent 的任务说明 |

其中 `baseline_timeline` 是任务单的核心字段。新闻 Agent 和金融 Agent 都应围绕这条时间线工作，并尽量把输出结果挂回对应的 `timeline_item_id`。

但“时间线驱动”不能理解成“只能查时间线里已经写出来的节点”。基准时间线来自百科概览，天然可能漏掉重要日期、外交回应、伤亡更新、官方表态、停火谈判、制裁动作或其他影响事件走向的材料。因此，新闻 Agent 应采用两段式采集：

| 采集阶段 | 作用 | 是否绑定 timeline_item_id |
| --- | --- | --- |
| Timeline Pass | 围绕 `baseline_timeline` 中已有点状事件、区间事件和背景项采集材料 | 是 |
| Discovery Pass | 围绕 `confirmed_event` 和 `confirmed_scope` 做受控补漏，发现基准时间线之外的重要材料 | 初始可以不绑定 |

Discovery Pass 是这套机制的“救场通道”。它允许新闻 Agent 发现时间线之外的重要信息，但不允许突破用户确认的事件范围，也不允许绕过来源质量限制。也就是说，Discovery Pass 仍然必须受 `confirmed_scope`、`exclude_scope` 和 P0-P3 `source_registry` 约束；不在注册表里的来源仍然默认丢弃。

如果 Discovery Pass 找到的材料可以挂到已有时间项，就应补挂到对应 `timeline_item_id`。如果材料明显属于本轮事件范围、来源合格、但无法对应任何已有时间项，新闻 Agent 不应擅自改写基准时间线，而应输出 `timeline_update_suggestions`，并写入事件基础信息库。第一版只保存建议，不做最终采纳。

第一版不需要在任务单中提前写复杂字段，例如完整市场观察列表、详细纳入规则、详细背景规则或最终输出 schema。这些可以在真实实现中逐步补充。原先的“采集计划”可以理解为这个任务单的简化名称。更准确地说，模块一需要的是一份“基于时间线的采集任务单”，而不是一张孤立字段表。

## 七、新闻 Agent 设计

新闻 Agent 负责回答：

```text
现实中发生了什么？
```

新闻 Agent 接收事件接入与任务分发 Agent 生成的时间线驱动新闻任务，而不是直接接收用户原始输入。它不应该泛泛搜索整个事件，而是先围绕基准事件时间线逐个时间项采集新闻、官方声明和公开资料，并把材料尽量挂回对应的 `timeline_item_id`。

同时，新闻 Agent 不能被基准时间线完全锁死。它还需要做一次受控的 Discovery Pass：围绕用户确认后的事件范围进行补漏，发现基准时间线可能漏掉的重要日期、声明、行动或阶段变化。Discovery Pass 的结果不能直接进入最终主时间线，只能作为来源材料和时间线修正建议进入事件基础信息库。

新闻 Agent 第一版建议采用“代码编排器 + LLM 判断器”的混合结构。代码负责生成查询、采集来源、抓取正文、去重、保存原始材料和结构校验；LLM 只负责相关性判断、事实摘要、来源差异整理和新闻事件块生成。这样可以避免让 LLM 自由上网搜索，也能避免把所有判断都写死在规则里。

### 1. Source Collector

新闻 Agent 内部应有一个 Source Collector。它是真正负责“像爬虫一样找材料”的部分，但采集顺序应按来源等级进行，而不是依赖单一新闻 API。

| 来源级别 | 来源类型 | 作用 |
| --- | --- | --- |
| P0 | 官方公告、政府网站、国际组织 | 一手材料、官方声明、政策文件、行动公告 |
| P1 | Reuters、AP、BBC、新华社、Al Jazeera 等高质量新闻源 | 高质量事实报道和跨来源确认 |
| P2 | 当地媒体、地区媒体 | 补充现场细节、地区视角和本地反应 |
| P3 | 金融媒体、智库、研究机构 | 补充影响分析、背景解释、政策或市场解读 |

P0 不等于“绝对事实”，而是“一手声明或官方立场”。例如政府公告可以证明某政府发布了某个声明，但不一定证明声明中的全部指控都是真的。P3 也不是低质量来源，只是它更适合解释影响，而不适合单独确认事件事实。

对每个时间线节点，Source Collector 可以按下面顺序采集：

```text
1. 优先查 P0 官方来源；
2. 再查 P1 高质量新闻；
3. 再查 P2 当地或地区媒体；
4. 再查 P3 金融媒体、智库和研究机构。
```

如果 P0 和 P1 已经提供了足够的高质量来源，系统可以减少低优先级来源采集，避免引入大量转载噪音。

### 2. 可实现流程

新闻 Agent 第一版可以按下面流程实现：

```text
1. 接收时间线驱动新闻任务；
2. 执行 Timeline Pass：对每个 timeline_item 生成搜索 query；
3. Source Collector 按 P0-P3 来源优先级采集候选 URL；
4. Fetcher 抓取网页正文和元数据；
5. Deduper 对 URL 和正文 hash 去重；
6. Relevance Judge 判断文章是否属于该时间项；
7. 执行 Discovery Pass：围绕 confirmed_event 和 confirmed_scope 补充搜索；
8. Relevance Judge 判断 Discovery 材料是否属于本轮事件范围、是否能挂到已有时间项；
9. News Block Builder 生成 news_blocks；
10. 对无法挂到已有时间项但明显重要的材料生成 timeline_update_suggestions；
11. Storage Writer 保存 source_documents、news_blocks 和 timeline_update_suggestions。
```

Source Collector 可以使用搜索引擎、站内搜索、RSS、网页读取和正文抽取等工具，但工具只是采集手段。新闻 Agent 的核心仍然是按来源等级选择材料、判断材料是否属于当前时间项、保留原始引用，并避免把低质量转载当作核心证据。

第一版应维护一个可配置的 `source_registry`，记录来源域名、来源等级、来源类型、地区、语言和是否允许作为核心证据。搜索引擎只能用于发现候选 URL；如果候选 URL 不在 P0-P3 的 `source_registry` 中，默认直接丢弃，不进入事件基础信息库。模块一优先保证信息层干净，不为了召回率引入无法评级的博客、转载站或未知来源。

如果 P0-P3 来源抓取正文失败，系统仍可以保存 URL、标题、发布时间、发布机构、抓取失败原因和来源等级，但不能把未抓到正文的材料用于生成事实摘要。这样可以保留引用线索，同时避免抓取失败的页面污染新闻事件块。

### 3. 输出结构

新闻 Agent 的输出不应只是一堆网页，而应至少包含三类结果：

| 输出 | 生成方 | 说明 |
| --- | --- | --- |
| source_documents | 新闻 Agent | 原始来源材料，包括 URL、标题、发布时间、来源等级、正文路径和引用 ID |
| news_blocks | 新闻 Agent | 围绕时间线节点整理出的结构化新闻事件块 |
| timeline_update_suggestions | 新闻 Agent | 新闻采集过程中发现基准时间线遗漏、日期错误或需要合并时给出的修正建议 |

新闻 Agent 不直接修改基准时间线，只能输出 `timeline_update_suggestions`。第一版只把这些建议保存到事件基础信息库，后续版本再决定是否采纳。

`source_documents` 可以包含：

| 字段 | 说明 |
| --- | --- |
| source_id | 来源 ID |
| event_id | 事件 ID |
| timeline_item_id | 对应的时间线节点或区间 |
| url | 来源链接 |
| title | 来源标题 |
| publisher | 发布机构或媒体 |
| published_at | 发布时间 |
| fetched_at | 抓取时间 |
| language | 语言 |
| source_tier | P0、P1、P2 或 P3 |
| source_type | official、wire、international_media、local_media、financial_media、think_tank 等 |
| evidence_role | fact_reporting、official_statement、analysis、background、lead 等 |
| discovery_method | 发现方式，例如 official_site_search、p1_source_search、rss、manual_source_list |
| collection_pass | timeline_pass 或 discovery_pass |
| content_hash | 正文 hash，用于去重 |
| raw_text_path | 原始正文保存路径 |
| fetch_status | 正文抓取状态，例如 success、metadata_only、failed |
| fetch_error | 抓取失败原因 |
| relevance_score | 与当前时间项的相关性分数 |

`news_blocks` 可以包含：

| 字段 | 说明 |
| --- | --- |
| news_block_id | 新闻事件块 ID |
| event_id | 事件 ID |
| timeline_item_id | 对应的时间线节点或区间 |
| title | 事件标题 |
| summary | 事件摘要 |
| event_time | 事件发生时间 |
| locations | 涉及地区 |
| actors | 涉及主体 |
| reported_facts | 来源材料中出现的事实性描述，不直接等同于系统最终确认事实 |
| source_summaries | 不同来源的报道摘要 |
| source_differences | 不同来源之间的差异、矛盾或叙事区别 |
| source_refs | 来源引用 |
| source_tier_summary | 该新闻块引用的 P0-P3 来源数量概览 |

新闻块里的 `reported_facts` 只是对来源材料的结构化摘录，不承担最终事实裁判功能。来源之间的报道差异、材料缺口和需要补采的地方，第一版先保留在 `source_differences`、`timeline_update_suggestions` 和来源引用中，后续版本再整理成更高层视图。

`timeline_update_suggestions` 可以包含：

| 字段 | 说明 |
| --- | --- |
| suggestion_id | 建议 ID |
| event_id | 事件 ID |
| suggested_time_type | point 或 interval |
| suggested_start_date | 建议新增或修正的开始日期 |
| suggested_end_date | 建议新增或修正的结束日期；点状事件可以为空 |
| title | 建议节点标题 |
| reason | 为什么认为时间线需要调整 |
| supporting_sources | 支持该建议的来源 ID |
| suggested_action | add_timeline_item、merge_timeline_item 或 adjust_date |

第一版新闻 Agent 可以只围绕用户输入事件进行一次性收集，不需要先实现持续监控。

## 八、金融 Agent 设计

金融 Agent 负责回答：

```text
市场如何理解这件事？
```

金融 Agent 接收事件接入与任务分发 Agent 生成的时间线驱动金融任务，并围绕同一个事件时间线整理市场证据。

对于“美伊冲突”这类事件，金融 Agent 同样必须基于用户确认后的事件范围和基准事件时间线工作。它既要观察关键日期附近的市场反应，也要识别跨多天或多周形成的市场趋势。

例如用户确认范围为某个具体美伊冲突事件后，金融 Agent 的任务可以是：

```text
请围绕已确认事件范围和基准事件时间线整理市场信号。

重点关注：
1. 关键点状事件前后的市场反应；
2. 区间事件内持续形成的市场趋势；
3. 原油、黄金、美元、美债、美股、地区股市、国防股、航运等相关资产；
4. 预测市场是否存在相关问题；
5. 历史上类似事件升级时的市场反应；
6. 哪些信号只是短期波动，哪些可能代表风险重新定价；
7. 金融总结和复盘材料如何解释当时市场反应。

范围约束：
1. 不把不属于确认范围的相似事件市场反应当作本轮事件反应；
2. 其他时间范围的相似事件只能作为类比或背景；
3. 类比材料必须和本轮事件材料分开标注；
4. 点状市场反应应挂到具体 timeline_item_id；
5. 跨多日形成的市场趋势应挂到一个 interval 时间项，或生成 market_trend_span。

输出结构化金融证据，不要直接给投资建议，也不要写最终推演结论。
```

金融 Agent 的输出应是结构化金融证据。

金融证据可以包含：

| 字段 | 说明 |
| --- | --- |
| timeline_item_id | 对应的时间线节点或区间；趋势类信号可以对应 interval |
| market_channel | 市场通道，例如原油、黄金、美元、美股、预测市场 |
| signal_type | 信号类型，例如价格变化、概率变化、波动率变化、历史类比 |
| observed_change | 观察到的变化 |
| time_window | 对应时间窗口 |
| possible_interpretation | 可能说明什么 |
| limitation | 这个信号的局限性 |
| source_refs | 数据或材料来源 |

金融 Agent 不直接输出最终结论。它只提供后续 Agent 可以引用的市场证据，避免把单一价格变化或单一预测市场概率当成确定性判断。

对于跨日期的金融趋势，可以单独输出市场趋势区间：

| 字段 | 说明 |
| --- | --- |
| market_trend_span_id | 趋势区间 ID |
| start_date | 趋势开始日期 |
| end_date | 趋势结束日期 |
| market_channel | 市场通道 |
| trend_summary | 趋势摘要 |
| related_timeline_items | 可能相关的事件时间项 |
| possible_drivers | 可能驱动因素 |
| source_refs | 数据或材料来源 |

这样油价上涨、避险资产走强、航运风险溢价上升等持续性变化，就可以作为一段时间线区间展示，而不是被硬塞进某一天。

## 九、事件基础信息库设计

事件基础信息库是模块一的底层材料仓。

它保存新闻 Agent 和金融 Agent 的产出，同时保留原始材料和结构化摘要。它不是直接给用户阅读的最终报告，而是后续 Context Map、模块二编排、RAG 检索和引用溯源的基础。第一版到事件基础信息库落盘为止，不继续生成 Context Map。

第一版可以先把事件基础信息库设计成事件级材料包。每个事件对应一组材料，而不是一开始就做全局复杂知识库。

一个事件级材料包可以包含：

| 字段 | 说明 |
| --- | --- |
| event_id | 事件 ID |
| event_query | 用户原始输入 |
| confirmed_event | 用户确认后的事件名称 |
| confirmed_scope | 用户确认后的事件范围说明 |
| timeline_collection_task | 时间线驱动采集任务单 |
| baseline_timeline | 统筹 Agent 提取的基准事件时间线 |
| news_blocks | 新闻 Agent 输出的新闻事件块 |
| timeline_update_suggestions | 新闻 Agent 对基准事件时间线的修正建议 |
| financial_evidence | 金融 Agent 输出的金融证据 |
| market_trend_spans | 金融 Agent 输出的跨日期市场趋势 |
| source_documents | 原始新闻、网页、公告、报告等来源材料 |
| entities | 主体、组织、人物、国家、市场等实体 |
| locations | 地理位置 |
| evidence_refs | 证据引用索引 |

事件基础信息库必须保留原始来源。模块一不能只保存 Agent 总结后的摘要，否则后续报告和问答无法追溯。

## 十、第一版边界：不实现事件地图 Agent

事件地图 Agent 暂时从第一版移除。

第一版模块一的终点是事件基础信息库，而不是 Context Map。系统只需要把以下内容保存清楚：

1. 用户确认后的事件范围；
2. 统筹 Agent 提取的基准事件时间线；
3. 时间线驱动采集任务单；
4. 新闻 Agent 输出的 `source_documents`；
5. 新闻 Agent 输出的 `news_blocks`；
6. 新闻 Agent 输出的 `timeline_update_suggestions`；
7. 金融 Agent 的空占位结果或后续版本预留字段；
8. 可追溯的原始材料、正文路径和引用索引。

因此，第一版不做这些事情：

1. 不生成最终事件时间线；
2. 不判断核心节点、辅助节点或背景节点；
3. 不生成事件概览；
4. 不生成主体地图、来源地图、报道差异地图或材料缺口地图；
5. 不提供基于事件材料库的问答；
6. 不把事件基础信息库加工成模块二可直接消费的 Context Map。

事件地图 Agent 可以作为第二版模块一的升级目标。等事件基础信息库稳定后，再让它读取 `baseline_timeline`、`news_blocks`、`timeline_update_suggestions`、`financial_evidence` 和 `source_documents`，统一整理最终时间线、事件概览和 Context Map。

## 十一、以“美伊冲突”为例的运行流程

用户输入：

```text
美伊冲突
```

模块一运行流程如下：

1. 事件接入与任务分发 Agent 接收用户输入。
2. 它识别“美伊冲突”可能对应多个事件范围。
3. 它向用户展示候选范围，例如 2020 年苏莱曼尼事件、近期美伊紧张局势、伊朗核问题、霍尔木兹海峡相关冲突等。
4. 用户选择一个范围，或补充具体时间、地区、关键词。
5. 事件接入与任务分发 Agent 根据用户选择生成事件范围确认卡。
6. 用户检查确认卡。如果范围不对，用户可以修改；统筹 Agent 根据修改重新生成确认卡。
7. 只有用户确认事件范围后，统筹 Agent 才读取 Wikipedia、Wikidata、百科页面或其他概览资料，提取基准事件时间线。
8. 基准时间线默认按天组织，例如背景、某个具体日期的关键事件、连续多日行动、后续协议、停火、谈判破裂或封锁等。
9. 统筹 Agent 不再让用户单独确认时间线，而是把这条时间线作为新闻 Agent 和金融 Agent 的初始采集骨架。
10. 统筹 Agent 基于基准时间线生成新闻 Agent 任务。
11. 统筹 Agent 基于基准时间线保留金融 Agent 任务接口，第一版可以只生成空占位任务。
12. 新闻 Agent 先按时间线节点和区间收集多来源报道，并输出带 timeline_item_id 的新闻事件块。
13. 新闻 Agent 再执行 Discovery Pass，在已确认事件范围内补查基准时间线可能遗漏的重要材料。
14. 如果 Discovery Pass 发现重要新日期、新区间或时间线错误，新闻 Agent 输出时间线修正建议，而不是直接改基准时间线。
15. 金融 Agent 第一版返回空占位结果，不接真实市场数据。
16. 基准时间线、新闻事件块、时间线修正建议、金融占位字段和原始材料进入事件基础信息库。
17. 系统将事件基础信息库保存为事件级 JSON 材料包。
18. 第一版到这里结束：拿到完整、可追溯的事件基础信息库即为成功。

## 十二、第一版设计取舍

### 1. 先做用户触发，不做持续监控

第一版只需要围绕用户输入事件进行一次性采集和整理。不需要先实现实时新闻流、热点追踪或自动监控。

### 2. 先做事件级材料包，不做全局知识库

每次用户输入事件后，先生成一个事件级材料包。等系统稳定后，再考虑跨事件关联和长期知识库。

### 3. 先做事件基础信息库，不做 Context Map

第一版不实现事件地图 Agent，也不生成 Context Map。先把事件范围、基准时间线、新闻材料、时间线修正建议、来源正文和引用索引保存稳定。等事件基础信息库稳定后，再考虑 Context Map、GraphRAG 或基础材料问答。

### 4. 统筹 Agent 只做范围确认、基准时间线和任务分发

事件接入与任务分发 Agent 不应该变成全局 Planner。它有和用户交互的对话能力，但这个对话只服务于确定事件范围。用户确认范围后，它负责提取基准时间线和生成模块一内部任务，不负责模块二、模块三和模块四的工作。

基准时间线不需要再让用户单独确认。统筹 Agent 可以用百科类资料生成初始骨架；如果骨架有遗漏或错误，第一版由新闻 Agent 和金融 Agent 输出修正建议并写入事件基础信息库，暂不统一修正成最终时间线。

### 5. 原始材料必须保留

新闻事实、金融信号、时间线修正建议和结构化摘要都应能追溯到来源。可追溯性是模块一最重要的设计原则之一。

## 十三、第一版验收标准

模块一第一版可以用以下标准验收：

1. 用户输入一个事件后，统筹 Agent 能识别候选事件范围。
2. 如果存在多个候选范围，统筹 Agent 能向用户追问并让用户选择或补充。
3. 统筹 Agent 能返回事件范围确认卡，展示事件名称、事件内容、时间范围、地理范围、关键主体、关键词、排除范围和背景材料规则。
4. 用户没有确认事件范围前，新闻 Agent 和金融 Agent 不应开始执行。
5. 用户确认事件范围后，系统能从百科类概览资料中提取基准事件时间线。
6. 基准时间线能支持点状事件和区间事件两种表达。
7. 时间线驱动任务单能包含 `event_query`、`confirmed_event`、`confirmed_scope`、`baseline_timeline`、`exclude_scope`、`news_task` 和 `finance_task`。
8. 时间线驱动任务单能让新闻 Agent 围绕基准时间线执行任务，并为金融 Agent 保留后续扩展接口。
9. Source Collector 能按 P0-P3 来源注册表采集材料，不在注册表中的未知来源默认丢弃。
10. 新闻 Agent 能执行 Timeline Pass，按时间线节点和区间输出结构化新闻事件块。
11. 新闻 Agent 能执行 Discovery Pass，在确认事件范围内发现基准时间线之外的重要候选材料。
12. Discovery Pass 仍然只能使用 P0-P3 注册来源，未知来源默认丢弃。
13. Discovery Pass 找到的材料如果能挂到已有时间项，应补挂到已有 `timeline_item_id`；如果无法挂载但明显重要，应生成 `timeline_update_suggestions`。
14. 新闻事件块使用 `reported_facts` 记录来源中的事实性描述，不输出系统确认类事实字段或全局不确定性判断。
15. 新闻 Agent 能输出 `timeline_update_suggestions`，但不直接修改基准时间线。
16. 金融 Agent 第一版可以返回空占位结果，事件基础信息库仍保留 `financial_evidence` 和 `market_trend_spans` 字段。
17. 原始材料、基准时间线、时间线修正建议、金融占位字段和结构化摘要都能进入事件基础信息库。
18. 系统能把事件基础信息库保存为事件级 JSON 材料包。
19. 事件基础信息库中每条新闻块、时间线修正建议和结构化摘要都能追溯到 `source_refs`。
20. 第一版不生成 Context Map，也不实现事件地图 Agent。

## 十四、后续升级方向

模块一第一版跑通后，可以继续升级：

- 从用户触发扩展到系统主动监控热点事件；
- 接入 Wikipedia、Wikidata 或 MediaWiki API，稳定抽取事件概览和时间线结构；
- 从单事件材料包扩展到跨事件关联；
- 实现事件地图 Agent，读取事件基础信息库生成 Context Map；
- 从结构化 Context Map 扩展到 GraphRAG；
- 增加来源可信度评分；
- 增加多语言新闻聚合；
- 增加历史相似事件检索；
- 增强金融 Agent 对不同资产类别的分析能力；
- 支持 Context Map 随新材料更新；
- 支持事件版本管理；
- 支持模块一独立问答工作台。
