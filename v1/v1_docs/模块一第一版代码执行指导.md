# 模块一第一版代码执行指导

## 一、目标与范围

本文档用于指导模块一第一版的代码实现。它基于 `docs/模块一设计.md`，目标是把“现实基础信息层”落成一个可运行、可测试、可迭代的最小版本。

第一版重点实现三件事：

1. 用户确认事件范围后，生成基准事件时间线。
2. 新闻 Agent 围绕时间线采集权威来源材料，并通过 Discovery Pass 补漏基准时间线之外的重要材料。
3. 将基准时间线、新闻材料、时间线修正建议、原始来源和金融占位结果保存为事件基础信息库。

第一版暂不实现完整金融 Agent。金融 Agent 只保留接口占位和数据结构占位，不接真实市场数据、不做价格分析、不生成正式金融证据。

## 二、第一版总流程

```text
用户输入事件
  ↓
事件接入与任务分发 Agent
  ↓
用户确认事件范围
  ↓
基准事件时间线生成
  ↓
时间线驱动采集任务单
  ↓
新闻 Agent
  ↓
事件基础信息库
```

第一版运行时，金融 Agent 可以被跳过，或返回空结果：

```json
{
  "financial_evidence": [],
  "market_trend_spans": [],
  "status": "not_implemented"
}
```

### 1. 环境变量与 API Key

第一版需要补一个环境配置层。模块一会调用 LLM、搜索服务、网页抓取服务或其他外部 API，这些密钥不能写进代码、JSON 材料包或文档正文，只能放在本地 `.env` 或部署环境变量里。


### 2. Agent API 对话层

模块一里的 Agent 可以接 API 对话，但第一版不要让每个 Agent 各自直接读 API Key。推荐统一封装一个 `llm/agent_client.py`：

```python
class AgentClient(Protocol):
    def chat_text(self, messages: list[dict]) -> str:
        ...

    def chat_json(self, messages: list[dict], schema: type[BaseModel]) -> BaseModel:
        ...
```

调用关系：

```text
settings.py 读取 .env / 环境变量
        ↓
agent_client.py 初始化 LLM 客户端
        ↓
coordinator / timeline_builder / relevance_judge / block_builder 调用 AgentClient
        ↓
AgentClient 返回结构化结果
        ↓
Pydantic 校验
        ↓
JSON 文件落盘
```

`AgentClient` 第一版支持三种 wire API：

1. `chat_completions`：请求 `/v1/chat/completions`，兼容旧版 Chat Completions 风格。
2. `responses`：请求 `/v1/responses`，用于官方 OpenAI Responses API 或兼容它的第三方网关。`model_reasoning_effort` 会映射为 `reasoning.effort`，`disable_response_storage=true` 会映射为 `store=false`。
3. `anthropic_messages`：请求 `/v1/messages`，用于 Anthropic / Claude-compatible 第三方网关。系统消息会被转换成 Anthropic 的 `system` 字段，普通消息保留为 `messages`。

第一版至少需要这些 API 调用点：

| 调用点 | 是否必须 | 用途 |
| --- | --- | --- |
| Coordinator | 可选 | 模糊事件范围识别、生成确认卡 |
| Timeline Builder | 可选 | 从百科段落中抽取日期和事件项 |
| Relevance Judge | 必须 | 判断来源是否属于确认范围、是否匹配时间项 |
| News Block Builder | 必须 | 从来源正文生成结构化 `reported_facts`、摘要和报道差异 |
| Storage Writer | 必须 | 将事件基础信息库保存为 JSON 文件 |

密钥安全规则：

1. 不把 `OPENAI_API_KEY` 或其他 API Key 写入 `event_info_package.json`。
2. 不把 API Key 写入 `source_documents`、日志、错误信息或 prompt 记录。
3. `.env.example` 可以提交，`.env` 不能提交。
4. 如果 API 调用失败，错误信息只记录服务名、状态码和请求 ID，不记录密钥。
5. 测试环境使用 fake client 或 mock client，不调用真实付费 API。

搜索与抓取规则：

1. `MODULE1_NEWS_AGENT_MODE=llm_tools`：新闻 Agent 由 LLM 决定下一步调用 `rss_search`、`web_search`、`fetch_url` 等工具；Python 只负责执行工具、做来源白名单过滤和落盘。
2. `MODULE1_NEWS_AGENT_MODE=rule_based`：保留旧的规则工作流，主要用于测试或无模型环境。
3. `MODULE1_SEARCH_PROVIDER=rss`：从 `news_feeds.yaml` 里的权威 RSS/Atom 源发现候选新闻链接，不需要搜索 API Key，适合最新事件和近期事件。
4. `MODULE1_SEARCH_PROVIDER=brave`：调用 Brave Search API 做 `site:domain` 站内搜索，需要 `MODULE1_SEARCH_API_KEY`，适合历史事件和补漏。
5. `MODULE1_SEARCH_PROVIDER=rss_brave`：RSS 和 Brave 两种工具都可供 LLM 调用，结果统一进入 SourceRegistry 白名单过滤。
6. RSS、Brave 和后续搜索工具都只负责发现 URL，不能绕过 `source_registry.yaml`。未知域名直接丢弃。
7. 真实网页正文由 `fetch_url` 工具通过 `UrlLibFetcher` 抓取；抓取失败的材料可以保留为 failed 记录，但不能生成 `reported_facts`。

## 四、核心数据模型

第一版仍然使用 JSON 文件作为存储，不做数据库。这里说的 Pydantic 不是 ORM，也不是数据库层，而是代码里的结构校验层：Agent 输出先进入 Pydantic Model，校验字段、类型和必填项，通过后再序列化成 JSON 写入文件。

如果后端暂时不想引入 Pydantic，也可以用 `dataclass` / `TypedDict` 加 JSON Schema 校验替代。但第一版推荐 Pydantic，因为它能把 LLM 或采集器输出中的字段缺失、类型错误和非法枚举提前拦住，避免脏数据直接写进事件材料包。

### 1. 事件范围确认

```python
class EventScopeConfirmation(BaseModel):
    event_query: str
    confirmed_event: str
    confirmed_scope: str
    time_range: str | None = None
    locations: list[str] = []
    key_actors: list[str] = []
    exclude_scope: str | None = None
    background_rule: str | None = None
    user_confirmed: bool
```

说明：

- `event_query` 是用户原始输入。
- `confirmed_event` 是用户确认后的事件名称。
- `confirmed_scope` 是用户确认后的事件范围说明。
- `exclude_scope` 用来防止查入相似但不属于本轮范围的事件。
- `user_confirmed` 必须为 `true` 后，才能进入时间线和新闻采集流程。

### 2. 基准事件时间线

```python
class TimelineItem(BaseModel):
    timeline_item_id: str
    time_type: Literal["point", "interval", "background"]
    start_date: str | None = None
    end_date: str | None = None
    title: str
    summary: str
    actors: list[str] = []
    locations: list[str] = []
    source_hint: str | None = None
    search_keywords: list[str] = []
```

规则：

- 默认按天组织。
- 点状事件使用 `start_date`。
- 区间事件使用 `start_date` 和 `end_date`。
- 背景项使用 `time_type = "background"`，可以没有具体日期。
- `search_keywords` 只是起始检索线索，不是限制条件。

### 3. 时间线驱动采集任务单

```python
class TimelineCollectionTask(BaseModel):
    event_id: str
    event_query: str
    confirmed_event: str
    confirmed_scope: str
    baseline_timeline: list[TimelineItem]
    exclude_scope: str | None = None
    news_task: str
    finance_task: str | None = None
```

第一版只要求这些最小字段。不要在第一版过早加入复杂市场观察列表、完整纳入规则或最终输出 schema。

### 4. 来源材料

```python
class SourceDocument(BaseModel):
    source_id: str
    event_id: str
    timeline_item_id: str | None = None
    url: str
    title: str | None = None
    publisher: str | None = None
    published_at: str | None = None
    fetched_at: str
    language: str | None = None
    source_tier: Literal["P0", "P1", "P2", "P3"]
    source_type: str
    evidence_role: str
    discovery_method: str
    collection_pass: Literal["timeline_pass", "discovery_pass"] = "timeline_pass"
    content_hash: str | None = None
    raw_text_path: str | None = None
    fetch_status: Literal["success", "metadata_only", "failed"]
    fetch_error: str | None = None
    relevance_score: float | None = None
```

规则：

- 不在 P0-P3 `source_registry` 中的来源默认丢弃。
- 抓取正文失败时，可以保存 metadata，但不能用于生成新闻事实摘要。
- `published_at` 是文章发布时间，不等同于事件发生时间。
- `collection_pass` 用来区分材料来自时间线节点采集，还是来自补漏采集。

### 5. 新闻事件块

```python
class NewsBlock(BaseModel):
    news_block_id: str
    event_id: str
    timeline_item_id: str
    title: str
    summary: str
    event_time: str | None = None
    locations: list[str] = []
    actors: list[str] = []
    reported_facts: list[dict] = []
    source_summaries: list[dict] = []
    source_differences: list[dict] = []
    source_refs: list[str] = []
    source_tier_summary: dict[str, int] = {}
```

规则：

- `reported_facts` 只是来源材料中的事实性描述，不等于系统最终确认事实。
- 新闻事件块不输出系统确认类事实字段，也不输出全局不确定性字段。
- 报道差异、材料缺口和需要补采的地方第一版只保留在新闻块、时间线修正建议和来源引用中，不再整理成 Context Map。

### 6. 时间线修正建议

```python
class TimelineUpdateSuggestion(BaseModel):
    suggestion_id: str
    event_id: str
    suggested_time_type: Literal["point", "interval"] = "point"
    suggested_start_date: str | None = None
    suggested_end_date: str | None = None
    title: str

    supporting_sources: list[str]
    suggested_action: Literal[

```

新闻 Agent 不直接修改 `baseline_timeline`。它只能输出 `timeline_update_suggestions`。第一版只保存建议，不做最终采纳判断。

### 7. 事件材料包

```python
class EventInfoPackage(BaseModel):

    timeline_collection_task: TimelineCollectionTask
```

`baseline_timeline` 是统筹 Agent 生成的初始时间线。第一版不生成最终时间线，因此不设置 `timeline_items`。

## 六、事件接入与时间线生成

### 1. Coordinator

`coordinator.py` 负责：

1. 接收用户输入。
2. 识别候选事件范围。
3. 生成事件范围确认卡。
4. 等待用户确认。
5. 生成 `TimelineCollectionTask`。

第一版可以先人工或半自动确认事件范围，不必做复杂多轮对话系统。只要保证 `user_confirmed = true` 后才进入下一步。

### 2. Wiki Client

`wiki_client.py` 负责读取 Wikipedia / Wikidata / MediaWiki API 或百科页面。

第一版最小能力：

1. 根据 `confirmed_event` 搜索相关百科页面。
2. 读取页面标题、摘要、目录和正文段落。
3. 提取可能的时间线段落或含日期段落。
4. 返回原始页面引用。

注意：

- Wikipedia 只是初始骨架来源，不是最终事实来源。
- 如果没有合适百科页面，可以退化为从用户确认范围和少量 P0/P1 材料生成初始时间线。

### 3. Timeline Builder

`timeline_builder.py` 负责把百科内容整理成 `TimelineItem`。

实现规则：

1. 默认按天组织。
2. 有明确日期的内容生成 `point`。
3. 持续多天的行动或阶段生成 `interval`。
4. 事件开始前的历史材料生成 `background`。
5. 不判断核心/辅助节点。
6. 不判断关键事实。

伪代码：

```python
def build_baseline_timeline(scope: EventScopeConfirmation) -> list[TimelineItem]:
    page = wiki_client.find_event_page(scope.confirmed_event)
    sections = wiki_client.extract_sections(page)
    dated_items = extract_dated_items(sections)
    timeline = normalize_to_timeline_items(dated_items)
    return sort_timeline(timeline)
```

## 七、新闻 Agent 实现

### 0. 两段式采集策略

新闻 Agent 第一版不能只做“逐个时间线节点搜索”，否则基准时间线一旦漏节点，后续材料就会被系统性漏掉。第一版应实现两段式采集：

| 阶段 | 方法 | 目的 |
| --- | --- | --- |
| Timeline Pass | `collect_for_timeline_item(task, item)` | 围绕已有 `baseline_timeline` 节点采集材料，并尽量挂到具体 `timeline_item_id` |
| Discovery Pass | `collect_for_discovery(task)` | 围绕 `confirmed_event` 和 `confirmed_scope` 做受控补漏，发现基准时间线之外的重要候选材料 |

Discovery Pass 不是自由搜索。它必须满足四个约束：

1. 候选 URL 必须命中 P0-P3 `source_registry`。
2. 内容必须属于 `confirmed_scope`，不能落入 `exclude_scope`。
3. 如果能匹配已有 `timeline_item_id`，就补挂到已有节点。
4. 如果无法匹配已有节点，但来源合格且明显重要，只能生成 `timeline_update_suggestions`，不能直接改 `baseline_timeline`。

### 1. Query Builder

`query_builder.py` 根据 `TimelineItem` 生成查询。

输入：

```python
TimelineCollectionTask
TimelineItem
```

输出：

```python
list[str]
```

第一版 query 规则：

```text
confirmed_event + timeline_item.title + start_date
confirmed_event + actor + start_date
timeline_item.search_keywords
site:<registered_domain> + timeline title + date
```

不要只搜用户原始输入。

Discovery Pass 需要单独生成更宽但仍受范围约束的 query：

```text
confirmed_event + confirmed_scope 中的关键主体
confirmed_event + confirmed_scope 中的关键地区
confirmed_event + strike / attack / ceasefire / talks / sanctions / retaliation / blockade / agreement 等强事件词
site:<registered_domain> + confirmed_event
```

这些 query 的作用是补漏，不是扩大事件范围。结果仍然必须经过 `source_registry`、`exclude_scope` 和相关性判断。

### 2. Source Collector

`source_collector.py` 按 P0-P3 顺序收集候选 URL。

伪代码：

```python
def collect_for_timeline_item(
    task: TimelineCollectionTask,
    item: TimelineItem,
) -> list[CandidateSource]:
    candidates = []

    for tier in ["P0", "P1", "P2", "P3"]:
        domains = source_registry.domains_for_tier(tier)
        for domain in domains:
            queries = query_builder.build_site_queries(task, item, domain)
            results = search_provider.search(queries)
            for result in results:
                source_meta = source_registry.match(result.url)
                if not source_meta:
                    continue
                candidates.append(
                    CandidateSource.from_result(
                        result,
                        source_meta,
                        timeline_item_id=item.timeline_item_id,
                        collection_pass="timeline_pass",
                    )
                )

        if enough_sources(candidates):
            break

    return dedupe_candidates(candidates)


def collect_for_discovery(task: TimelineCollectionTask) -> list[CandidateSource]:
    candidates = []

    for tier in ["P0", "P1", "P2", "P3"]:
        domains = source_registry.domains_for_tier(tier)
        for domain in domains:
            queries = query_builder.build_discovery_queries(task, domain)
            results = search_provider.search(queries)
            for result in results:
                source_meta = source_registry.match(result.url)
                if not source_meta:
                    continue
                candidates.append(
                    CandidateSource.from_result(
                        result,
                        source_meta,
                        timeline_item_id=None,
                        collection_pass="discovery_pass",
                    )
                )

    return dedupe_candidates(candidates)
```

`enough_sources` 第一版可以简单设定：

```text
P0 >= 1 或 P1 >= 2 时，可减少低优先级采集。
```

这不是事实确认，只是控制采集规模。

Discovery Pass 不使用 `enough_sources` 提前停止得太快，因为它的目的不是给每个节点补齐材料，而是发现可能漏掉的节点。但第一版应设置总量上限，例如每个 source tier 最多保留前 10 个候选，避免搜索结果膨胀。

### 3. Fetcher 与 Extractor

`fetcher.py` 负责 HTTP 请求。  
`extractor.py` 负责正文抽取。

第一版规则：

1. 成功抓到正文：`fetch_status = success`。
2. 只抓到标题、发布时间、摘要：`fetch_status = metadata_only`。
3. 完全失败：`fetch_status = failed`。
4. `metadata_only` 和 `failed` 不用于生成 `reported_facts`。
5. 所有成功正文保存到 `data/module1/sources/<source_id>.txt`。

### 4. Deduper

`deduper.py` 负责：

1. URL 规范化去重。
2. 正文 hash 去重。
3. 同一媒体同一文章重复链接合并。

第一版不需要复杂语义去重。

### 5. Relevance Judge

`relevance_judge.py` 判断来源是否属于当前时间项。

第一版采用规则 + LLM 结构化判断：

规则过滤：

1. URL 来源必须在 `source_registry`。
2. 标题或正文需要命中事件名、时间项标题、关键主体或关键词。
3. 发布时间不能明显落在事件范围之外，除非是复盘材料。
4. Discovery Pass 材料必须落在 `confirmed_scope` 内，且不能命中 `exclude_scope`。

LLM 输出必须是结构化 JSON：

```json
{
  "is_relevant": true,
  "timeline_item_id": "tl_001",
  "matched_existing_timeline_item_id": "tl_001",
  "is_in_confirmed_scope": true,
  "needs_timeline_update_suggestion": false,
  "confidence": 0.82,
  "reason": "文章报道了该日期发生的事件，并提到同一批关键主体。"
}
```

如果 `confidence < 0.65`，第一版默认丢弃。

对于 Discovery Pass，判断顺序应是：

1. 先判断材料是否属于用户确认后的事件范围。
2. 再判断它是否能匹配已有 `timeline_item_id`。
3. 能匹配则补挂到已有时间项。
4. 不能匹配但明显重要，则进入 `TimelineUpdateSuggestion` 判断。
5. 不能匹配且重要性不足，则只保留为普通来源线索或直接丢弃，第一版建议丢弃。

“明显重要”的第一版规则可以先保守定义：

```text
满足以下任一条件，才允许生成 timeline_update_suggestions：
1. P0 来源直接发布新的官方行动、声明、制裁、协议或停火信息；
2. 至少两个 P1/P2 来源报道同一新增日期或新增行动；
3. 材料涉及 confirmed_scope 中的关键主体，并包含 strike、attack、ceasefire、sanction、retaliation、negotiation、blockade、agreement、collapse 等强事件词；
4. 材料有明确日期或可归一到明确日期区间。
```

### 6. News Block Builder

`block_builder.py` 负责把同一 `timeline_item_id` 下的来源材料整理成 `NewsBlock`。

规则：

1. 只使用 `fetch_status = success` 且相关性通过的来源。
2. 只生成 `reported_facts`，不生成系统确认类事实字段。
3. 对不同来源的明显差异写入 `source_differences`。
4. 所有摘要和事实性描述必须带 `source_refs`。
5. 不做最终事实裁判。
6. Discovery Pass 材料只有在补挂到已有 `timeline_item_id` 后，才进入对应 `NewsBlock`。
7. 无法挂到已有时间项的 Discovery 材料，不直接生成 `NewsBlock`，而是用于生成 `timeline_update_suggestions`。

### 7. Timeline Update Suggestions

新闻 Agent 在采集过程中如果发现：

1. 多个 P0/P1/P2 来源提到一个基准时间线没有的新日期；
2. 多个 P0/P1/P2 来源提到一个基准时间线没有的新日期区间；
3. 某个基准时间线日期明显错误；
4. 两个时间项实际描述同一件事；

则生成 `TimelineUpdateSuggestion`。

新闻 Agent 不能直接修改 `baseline_timeline`。

建议生成逻辑：

```python
def build_timeline_update_suggestions(
    discovery_sources: list[SourceDocument],
    baseline_timeline: list[TimelineItem],
) -> list[TimelineUpdateSuggestion]:
    grouped = group_discovery_sources_by_event_date_and_title(discovery_sources)
    suggestions = []

    for group in grouped:
        if can_match_existing_timeline_item(group, baseline_timeline):
            continue
        if not is_important_discovery_group(group):
            continue

        suggestions.append(
            TimelineUpdateSuggestion(
                suggestion_id=new_id("tus"),
                event_id=group.event_id,
                suggested_time_type=group.suggested_time_type,
                suggested_start_date=group.start_date,
                suggested_end_date=group.end_date,
                title=group.title,
                reason=group.reason,
                supporting_sources=group.source_ids,
                suggested_action="add_timeline_item",
            )
        )

    return suggestions
```

## 十一、端到端 Pipeline

`pipeline.py` 负责串联：

```python
def run_module1(event_query: str) -> EventInfoPackage:
    settings = load_module1_settings()
    agent_client = build_agent_client(settings)

    scope = coordinator.confirm_scope(event_query)
    assert scope.user_confirmed is True

    baseline_timeline = timeline_builder.build_baseline_timeline(
        scope,
        agent_client=agent_client,
    )
    task = coordinator.build_timeline_collection_task(scope, baseline_timeline)

    news_result = news_agent.run(task, agent_client=agent_client)
    finance_result = run_finance_placeholder(task)

    package = storage.build_event_info_package(
        scope=scope,
        task=task,
        baseline_timeline=baseline_timeline,
        news_result=news_result,
        finance_result=finance_result,
    )
    storage.save_event_package(package)

    return package
```
