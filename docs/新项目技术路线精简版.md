# 新项目技术路线精简版

## 1. 项目定位

本项目不是继续做 OASIS 的社交媒体模拟，而是搭建一个面向现实世界的多智能体态势推演平台。

原来的 OASIS 主要模拟社交媒体中的发帖、评论、点赞、关注等交互；新项目要模拟的是国家、组织、媒体、市场和公众群体在现实事件驱动下的行动、反应、冲突升级与风险变化。

因此，项目核心应从“社交平台环境”转向“现实世界状态推演引擎”。

## 2. 总体技术路线

系统按照五个模块建设：

1. 前端与可视化
2. 数据与事件输入
3. 中央控制与状态管理
4. 智能体生态系统
5. 输出、解释与持久化

推荐技术选型：

| 模块 | 推荐方案 | 说明 |
| --- | --- | --- |
| 前端可视化 | CesiumJS + ECharts / D3 | 用于 3D 地球、风险热力图、时间线和事件链展示 |
| 后端服务 | Python Web 后端 | 可用 FastAPI，若要复用 MiroFish 也可先沿用 Flask |
| 多智能体编排 | LangGraph | 负责推演流程、Agent 调度、状态节点流转 |
| 权威数据存储 | SQL 数据库 | 保存事件、世界状态、行动、风险、解释和回放数据 |
| 运行时状态 | Redis | 保存任务状态、当前 step、暂停/恢复控制和实时缓存 |
| 实时通信 | WebSocket | 推送推演进度、风险变化、Agent 行动和地图更新 |
| RAG | 可选后置 | 只作为历史背景和证据检索，不作为状态源 |

## 3. 核心架构

建议整体流程如下：

```text
用户输入事件
  -> 事件标准化与校验
  -> SQL 事件库
  -> 调度器启动推演
  -> LangGraph 中央调度
  -> Agent 观察局势并提出行动
  -> 裁判层校验行动
  -> 规则引擎更新世界状态
  -> 风险评分与解释生成
  -> SQL 持久化
  -> WebSocket 推送前端
  -> 前端 3D 地球与时间线展示
```

需要注意：世界状态本身不负责触发推演循环。推演循环应由独立的 Scheduler / Clock / Task Runner 驱动。

## 4. 模块一：前端与可视化

前端负责用户交互、3D 地球展示、风险变化展示和历史回放。

主要能力：

- 创建和选择推演场景
- 注入用户自定义事件
- 在 3D 地球上展示地区、国家、事件点和风险热力图
- 展示时间轴、Agent 行动流和风险曲线
- 展示每一步的解释链路
- 支持暂停、恢复、停止和历史回放

前端通信方式建议分为两类：

- REST API：用于创建场景、查询历史、读取回放、下载报告
- WebSocket：用于推演过程中的实时画面更新

## 5. 模块二：数据与事件输入

事件输入不能直接进入中央调度器，必须先经过标准化处理。

事件来源包括：

- 用户自定义事件
- 新闻事件
- 手动录入的场景事件
- 后续可扩展的外部数据源

事件标准化需要完成：

- 事件类型识别
- 参与方识别
- 时间与地点解析
- 地理编码
- 强度归一化
- 来源可信度标注
- 重复事件去重
- 结构化校验
- 写入 SQL 事件库

标准化后的事件应包含：

- 事件类型
- 参与方
- 目标方
- 发生地区
- 发生时间
- 事件强度
- 来源信息
- 置信度
- 简要描述

RAG 可以放在该模块中，但建议后置建设。RAG 的职责是提供历史背景、相似事件和证据材料，不应直接决定世界状态或风险分数。

## 6. 模块三：中央控制与状态管理

中央控制模块是系统核心，由三部分组成：

- LangGraph Orchestrator：负责流程编排和 Agent 调度
- Referee / 裁判层：负责行动校验、冲突裁决和结果约束
- Rule Engine：负责执行状态转移

一次推演 step 的基本流程：

1. 调度器触发新的时间步
2. 读取上一帧 World State
3. 读取本轮新增事件
4. 选择需要参与的 Agent
5. Agent 输出行动提案
6. 裁判层校验行动是否合法
7. 规则引擎计算状态变化
8. 生成下一帧 World State
9. 计算风险评分
10. 生成解释
11. 持久化并推送前端

World State 是现实世界状态快照，应保存国家、地区、关系、风险因子和事件影响。它是系统最重要的数据结构之一，必须持久化到 SQL，不能只存在 Redis 或 LangGraph 内存中。

Redis 只用于运行时控制，例如：

- 当前推演状态
- 当前 step
- 暂停、恢复、停止指令
- WebSocket 推送缓存
- 短期任务进度

## 7. 模块四：智能体生态系统

新项目中的 Agent 不再是社交媒体用户，而是现实世界中的行动主体。

建议先支持以下类型：

- 国家 Agent
- 国际组织 Agent
- 媒体 Agent
- 市场 Agent
- 公众群体 Agent

每个 Agent 应有自己的角色设定：

- 战略目标
- 能力范围
- 约束条件
- 风险偏好
- 红线
- 盟友与对手
- 可选行动集合
- 决策风格

Agent 只能输出行动提案，不能直接修改世界状态。

行动提案可以包括：

- 外交声明
- 谈判请求
- 制裁
- 军事信号
- 联盟协调
- 舆论动员
- 市场反应
- 降级行动
- 保持观望

裁判层和规则引擎负责判断这些行动是否成立，以及它们如何改变世界状态。

## 8. 模块五：输出、解释与持久化

输出模块包括风险评分、解释生成、日志记录和回放存储。

风险评分建议先采用可解释的规则模型，而不是完全交给大模型判断。大模型可以用于解释和总结，但风险分数应主要来自结构化状态和规则计算。

风险评分可以考虑：

- 军事紧张度
- 外交沟通程度
- 敌对关系强度
- 事件强度
- 误判风险
- 联盟介入程度
- 公众关注度
- 经济压力

解释模块需要说明：

- 本轮发生了什么
- 哪些 Agent 采取了行动
- 哪些行动被采纳或拒绝
- 世界状态发生了哪些变化
- 风险为什么上升或下降

所有输出都必须持久化，不能只推送给前端。

需要保存的数据包括：

- 每轮输入事件
- 每轮世界状态
- Agent 行动提案
- 裁判层裁决结果
- 规则引擎状态变化
- 风险评分
- 解释文本
- 回放日志

这样系统才能支持历史回放、报告生成和结果复现。

## 9. 与 MiroFish 的关系

MiroFish 可以参考，但不能直接作为新项目内核。

可参考的部分：

- 项目管理流程
- 文件上传与任务状态管理
- 图谱构建思路
- Agent Profile 生成思路
- 行动日志记录方式
- 报告生成流程

需要替换的部分：

- OASIS 社交媒体模拟环境
- 发帖、评论、点赞、关注等社交动作
- Twitter / Reddit 风格数据库结构
- 以社交互动为中心的仿真逻辑

新项目的核心应是：

```text
现实事件 -> 世界状态 -> Agent 行动提案 -> 裁判与规则引擎 -> 新世界状态 -> 风险与解释
```

## 10. 建设阶段

### 第一阶段：最小闭环

目标是先让系统跑起来。

建设内容：

- 场景创建
- 事件录入
- 简单 World State
- 简单 LangGraph 推演流程
- 简单规则引擎
- 风险分数计算
- WebSocket 推送
- 前端展示风险点和时间线

验收标准：

- 用户能创建场景
- 用户能输入事件
- 系统能按 step 推演
- 前端能看到风险变化
- 历史结果可以回放

### 第二阶段：事件标准化

目标是让输入事件变得结构化、稳定、可计算。

建设内容：

- 新闻和用户文本解析
- 参与方识别
- 地区识别
- 事件强度评分
- 来源可信度标注
- SQL 事件库

### 第三阶段：现实世界 Agent

目标是替换 OASIS 的社交媒体 Agent。

建设内容：

- 国家 Agent
- 组织 Agent
- 媒体 Agent
- 市场 Agent
- Agent Profile 管理
- 行动提案格式约束

### 第四阶段：规则引擎和风险模型

目标是让推演结果更可解释、更稳定。

建设内容：

- 行动合法性校验
- 状态转移规则
- 冲突裁决逻辑
- 风险评分模型
- 风险驱动因子解释

### 第五阶段：解释、回放和报告

目标是让推演结果能被理解、复现和汇报。

建设内容：

- 每一步解释生成
- 风险变化原因分析
- 事件链展示
- 历史回放
- 推演报告生成

### 第六阶段：RAG 历史证据层

目标是增强 Agent 决策和解释的背景依据。

建设内容：

- 历史资料上传
- 文本解析
- 实体与事件抽取
- 向量索引
- 相似事件检索
- 证据引用

注意：RAG 应作为辅助证据层，而不是系统状态源。

## 11. 关键原则

1. Agent 只提出行动，不直接修改状态。
2. 世界状态必须结构化，并持久化到 SQL。
3. Redis 只保存运行时状态，不作为权威数据库。
4. RAG 只提供历史背景和证据，不决定推演结果。
5. 风险评分要可解释，不应完全由大模型自由生成。
6. 每一步推演都必须可回放、可追踪、可复现。
7. 先跑通最小闭环，再逐步增加 RAG、复杂 Agent 和高级可视化。

## 12. 一句话总结

本项目的技术路线不是“把 OASIS 换成 LangGraph”，而是用 LangGraph 搭建一个现实世界态势推演流程：由事件驱动世界状态变化，由 Agent 提出行动，由裁判和规则引擎更新状态，再通过风险评分、解释生成和 3D 可视化呈现局势演化。
