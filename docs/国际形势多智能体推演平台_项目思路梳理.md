# 国际形势多智能体推演平台：项目思路梳理

## 1. 一句话定位

本项目要做的不是普通聊天机器人，也不是简单新闻分析工具，而是一个面向国际形势的多智能体态势推演平台。

用户提供项目背景、现实新闻、重大事件和推演目标后，系统构建若干具有现实角色约束的 Agent，让它们在同一个局势沙盘中观察、判断、行动和相互影响，最终生成一条可解释、可回放、可分析的局势演化过程。

它的目标不是准确预测未来，而是帮助用户看到：

- 不同国际主体可能如何理解同一事件
- 不同行动会如何影响局势走向
- 哪些因素可能导致风险升级或降级
- 哪些视角容易被人忽略
- 当前事件在历史和现实背景中可能意味着什么

因此，本项目更接近“现实世界态势沙盘 + 多智能体推理实验平台”，而不是确定性的预测系统。

## 2. 项目背景

国际形势分析天然复杂。一个事件往往同时牵涉国家利益、历史矛盾、联盟关系、地理位置、军事能力、经济压力、国内政治、媒体舆论和公众情绪。

传统分析工具通常有几个问题：

- 只能做静态总结，难以模拟后续演化
- 只能从单一视角分析，缺少多方互动
- 依赖人工经验，难以系统化复盘
- 很难把海量新闻、历史事件和角色背景组织成可推演结构

多智能体系统提供了一种新的可能：把不同国际主体抽象成不同 Agent，让它们基于各自的目标、约束和信息环境进行判断，再由系统裁判层和规则引擎把这些判断转化为局势变化。

这个方向很新，成熟参考较少。MiroFish 可以作为工程上的借鉴，但它和本项目的核心目标不同。

## 3. 与 MiroFish 的区别

MiroFish 是一个基于社交媒体环境的通用多 Agent 推演系统。它的核心是让 Agent 在类似 Twitter / Reddit 的环境中发帖、评论、点赞和互动，从而观察社交舆论传播和群体行为。

本项目要做的是国际形势的现实沙盘。它关注的不是社交平台上的内容互动，而是现实世界中的国家、组织、媒体、市场和公众群体在事件驱动下如何行动，以及这些行动如何改变局势。

二者的区别可以概括为：

| 维度 | MiroFish | 本项目 |
| --- | --- | --- |
| 推演环境 | 社交媒体平台 | 现实世界态势沙盘 |
| Agent 类型 | 社交媒体用户 | 国家、组织、媒体、市场、公众群体 |
| 行动类型 | 发帖、评论、点赞、关注 | 外交、军事、经济、舆论、市场反应 |
| 状态对象 | 社交平台状态 | 世界状态、地区状态、主体关系、风险状态 |
| 输出重点 | 舆论传播与互动 | 局势演化、风险变化、事件链解释 |
| 工程价值 | 可参考任务流、日志、报告、Profile 生成 | 需要重建推演内核 |

因此，本项目不能简单复用 OASIS 的社交模拟内核，而应将其替换为现实世界状态推演引擎。

## 4. 核心问题

本项目最难的问题不是“调用多个大模型”，而是如何让多个 Agent 在有限上下文中掌握足够多、足够准确、足够相关的背景信息，并围绕同一事件进行稳定推演。

核心难点包括：

1. 背景信息巨大  
   国际形势涉及历史事件、地缘关系、国家利益、条约、政策、新闻和舆论，很难一次性塞进模型上下文。

2. Agent 视角不同  
   同一事件在不同国家、组织、媒体或市场主体眼中含义不同，Agent 不能共享同一种“万能视角”。

3. 推演需要动态状态  
   事件不是一次性分析，而是一步步改变世界状态。系统必须保存每一步的状态变化。

4. LLM 输出不可完全信任  
   大模型适合生成判断、解释和行动提案，但不适合直接决定事实、状态和风险分数。

5. 结果需要可追踪  
   用户需要知道为什么某个 Agent 这么判断，为什么风险上升，哪些背景材料支持了这个推理。

所以，本项目的关键不是让 Agent “自由聊天”，而是让 Agent 在结构化背景、角色约束、证据材料和世界状态之中进行受控推理。

## 5. 基本方法论

本项目应采用“结构化状态 + 证据检索 + Agent 提案 + 裁判规则 + 可解释输出”的方法。

核心原则是：

```text
事实由数据库保存
背景由检索系统提供
状态由 World State 表示
Agent 只提出行动和判断
裁判层负责校验
规则引擎负责状态变化
风险模型负责评分
解释模块负责说明原因
```

这种设计可以避免系统完全依赖大模型自由发挥。

大模型应该承担这些任务：

- 理解事件含义
- 生成 Agent 视角下的判断
- 提出可能行动
- 总结局势变化
- 生成解释文本
- 提供新颖视角

大模型不应该直接承担这些任务：

- 保存事实
- 保存长期状态
- 直接修改世界状态
- 直接决定风险分数
- 直接覆盖裁判层和规则引擎

## 6. 信息如何组织

为了让 Agent 掌握海量背景信息，不能简单把所有新闻和材料放进 prompt。更合理的方法是把信息分层。

建议将信息分为六层：

### 6.1 项目背景层

描述本次推演的大背景，例如：

- 推演主题
- 地区范围
- 时间范围
- 主要矛盾
- 用户关注的问题
- 推演目标

这部分对所有 Agent 可见。

### 6.2 Actor 背景层

描述每个现实主体的长期特征，例如：

- 国家利益
- 战略目标
- 能力范围
- 风险偏好
- 红线
- 盟友与对手
- 历史立场
- 国内约束

这部分构成 Agent Profile，是 Agent 角色稳定性的来源。

### 6.3 现实事件层

保存当前输入的新闻、大事和用户自定义事件。事件需要经过标准化，不能只保留自然语言原文。

每个事件至少应包含：

- 事件类型
- 参与方
- 地区
- 时间
- 强度
- 来源
- 置信度
- 摘要

### 6.4 历史证据层

保存历史新闻、相似事件、政策文件、国际关系背景和参考材料。

这一层可以使用 RAG，但 RAG 的定位是“证据检索”，不是“状态存储”。

### 6.5 世界状态层

表示当前沙盘中的局势状态，例如：

- 地区军事紧张度
- 外交沟通程度
- 经济压力
- 舆论关注度
- 国家间敌对程度
- 联盟介入程度
- 冲突概率

World State 是推演的核心状态，必须结构化并持久化。

### 6.6 推演记忆层

保存过去每一步发生了什么：

- Agent 看到了什么
- Agent 提出了什么行动
- 哪些行动被采纳
- 状态如何变化
- 风险为什么变化
- 系统生成了什么解释

这部分用于回放、复盘和后续 step 的上下文。

## 7. Agent 应该如何工作

Agent 不是自由聊天角色，而是受约束的现实主体模拟器。

每个 Agent 在每一轮推演中应接收：

- 当前项目背景
- 自己的 Actor Profile
- 当前 World State
- 本轮相关事件
- 与自己有关的历史证据
- 最近几轮关键变化
- 系统规定的可选行动类型

Agent 输出的不是“最终局势变化”，而是 Action Proposal。

Action Proposal 应包括：

- 行动主体
- 行动类型
- 目标对象
- 目标地区
- 行动强度
- 行动理由
- 预期影响
- 不确定性说明
- 引用的背景证据

关键约束：

```text
Agent 只能提出行动提案，不能直接修改世界状态。
```

这样设计的原因是：LLM 可以模拟判断，但不能拥有最终裁决权。最终状态变化必须由裁判层和规则引擎统一处理。

## 8. 多 Agent 推演流程

一次推演 step 可以理解为：

```text
读取当前局势
  -> 分发相关信息给 Agent
  -> Agent 分别判断并提出行动
  -> 系统收集行动提案
  -> 裁判层校验和冲突裁决
  -> 规则引擎更新世界状态
  -> 风险评分模块计算风险
  -> 解释模块生成事件链说明
  -> 保存并推送结果
```

多 Agent 的价值不在于让它们随意辩论，而在于让不同角色在同一局势下产生差异化判断。

例如同一个军事演习事件：

- 国家 Agent 可能关注威慑、联盟和红线
- 国际组织 Agent 可能关注降级机制和斡旋空间
- 媒体 Agent 可能关注舆论扩散和叙事框架
- 市场 Agent 可能关注能源、航运和资本市场风险
- 公众群体 Agent 可能关注民族主义、恐慌或抗议情绪

这种多视角差异，是项目可能产生新洞察的地方。

## 9. RAG 的合理位置

RAG 在本项目中有价值，但不能被过度神化。

它适合解决：

- 给 Agent 提供相关历史背景
- 查找类似事件
- 提供政策、条约、新闻证据
- 支持解释模块引用来源
- 辅助报告生成

它不适合解决：

- 直接保存当前状态
- 直接决定下一步局势
- 直接生成权威事实
- 直接给出风险评分

所以 RAG 应被设计为“材料检索层”，而不是“推演核心”。

更合理的运行方式是：

```text
当前事件 + Agent 身份 + 当前 World State
  -> 检索相关背景材料
  -> 组成 Agent Context Pack
  -> Agent 基于材料输出行动提案
```

系统应尽量给 Agent “少量但高价值”的材料，而不是大量无筛选材料。

## 10. 我们真正想得到什么

本项目不应该承诺“准确预测国际局势”。这是不现实的，也容易被质疑。

更合理的目标是：

1. 生成多方视角  
   展示不同主体可能如何理解同一个事件。

2. 生成可能行动  
   展示各方在不同约束下可能采取哪些行动。

3. 生成演化路径  
   给出若干步之后局势可能如何升级、降级或转向。

4. 发现风险驱动因素  
   找出导致风险变化的关键因素。

5. 提供新颖分析角度  
   让用户看到一些人工分析中可能遗漏的链条或视角。

6. 支持复盘和解释  
   每一步都能说明依据、行动、状态变化和风险原因。

因此，项目的价值不是“告诉用户未来一定会发生什么”，而是“帮助用户更系统地探索未来可能性”。

## 11. 判断系统好坏的标准

系统是否有效，不应只看输出是否像分析文章，而应看以下指标：

- 角色一致性：Agent 的行动是否符合自身背景和约束
- 证据相关性：Agent 使用的参考材料是否和事件相关
- 状态连续性：每一步世界状态是否承接上一轮
- 行动合理性：行动是否符合主体能力和现实约束
- 风险可解释性：风险变化是否能被结构化因素解释
- 多视角差异：不同 Agent 是否真的提供了不同判断
- 回放完整性：历史推演是否可以完整复现
- 新颖性：是否提供了有启发的新角度

这些标准比“模型说得像不像专家”更重要。

## 12. 第一版应该怎么收敛

第一版不要追求完整国际关系系统，也不要一开始接入海量新闻。

建议第一版只做一个小而完整的闭环：

```text
一个地区
三到五个 Agent
少量结构化事件
十到二十个推演 step
可解释风险分数
前端时间线和地图展示
完整回放
```

第一版重点验证：

- Agent 是否能基于背景产生差异化行动
- 行动是否能通过规则改变世界状态
- 风险是否能随状态变化
- 用户是否能理解每一步为什么发生

在这个闭环成立之前，不建议投入过多精力做复杂 RAG、自动新闻抓取、大规模 Agent 或高复杂度可视化。

## 13. 后续文档拆分

这份文档解决的是项目思想和定位。后续应继续拆成几份更具体的设计文档：

1. MVP 范围设计  
   明确第一版做什么、不做什么。

2. 领域模型设计  
   定义 Scenario、Actor、AgentProfile、Event、WorldState、Action、RiskFrame 等核心对象。

3. 推演流程设计  
   定义一次 step 如何从事件输入走到状态变化。

4. Agent 设计  
   定义 Agent 类型、Profile 字段、输入上下文和输出格式。

5. World State 设计  
   定义现实世界状态由哪些维度组成。

6. 规则引擎与风险评分设计  
   定义行动如何改变状态，风险如何计算。

7. 数据存储与回放设计  
   定义哪些数据进入 SQL、哪些进入 Redis、如何支持回放。

8. 前端交互设计  
   定义用户如何创建场景、注入事件、观察推演和复盘结果。

## 14. 最核心的判断

这个项目真正要解决的不是“让多个 Agent 聊天”，而是：

```text
如何把国际形势中的海量背景信息、现实事件、角色立场和状态变化组织成一个可推演、可解释、可回放的系统。
```

MiroFish 给了一个工程上的起点，但本项目要走向的是现实世界沙盘。

系统的关键能力不是让 Agent 说很多话，而是让它们在有限但高价值的背景材料中，基于明确角色和当前世界状态，提出合理、差异化、可追踪的行动判断。然后由裁判层、规则引擎和风险模型把这些判断转化为结构化的局势演化。

这是一个探索性很强的方向。它不应该被包装成确定性预测机器，而应该被定义为一个帮助用户探索复杂国际局势可能性的智能推演平台。
